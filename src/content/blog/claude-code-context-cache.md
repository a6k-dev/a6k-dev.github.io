Claude Code로 긴 세션을 돌리다 보면 문득 궁금해지는 순간이 있습니다. "모델은 매 턴마다 전체 대화를 통째로 다시 받는다는데, 왜 요금은 그만큼 안 나오지?" 그 답이 바로 **Context Cache**, 정확히는 Anthropic API의 **Prompt Caching(프롬프트 캐싱)** 입니다.

Claude Code는 이 캐시를 **자동으로** 관리합니다. 우리가 직접 신경 쓸 일은 거의 없지만, 원리를 알면 "왜 갑자기 세션이 느려지고 비싸졌지?" 같은 순간을 이해하고 피할 수 있습니다. 이 글에서는 캐시가 어떻게 동작하는지, 무엇이 캐시를 살리고 무엇이 깨뜨리는지를 하나하나 들여다봅니다. 캐시가 잘 맞으면 응답이 빨라지고 입력 토큰 비용이 **최대 90%** 까지 줄어듭니다.

> 🧊 이 글은 Claude Code 공식 문서와 Anthropic 엔지니어링 블로그를 1차 자료로 삼아 정리했습니다. 퍼센트 절감 수치 중 일부는 써드파티 가이드의 추정치이며, 그런 경우 본문에 표시했습니다. (정리 기준: 2026-06-22)

## 0. 한눈 요약 (TL;DR)

| 핵심 | 내용 |
| --- | --- |
| **무엇** | 매 턴 다시 보내는 컨텍스트의 "앞부분(prefix)"을 서버가 기억했다가 재사용 |
| **이득** | 캐시 읽기 = 표준 입력가의 **10%** (90% 할인) |
| **비용** | 캐시 쓰기 = 5분 TTL **1.25x**, 1시간 TTL **2.0x** (1회성) |
| **수명** | 기본 5분, 옵션 1시간. 적중할 때마다 타이머 리셋 |
| **원리** | **정확 일치(exact) prefix 매칭.** 앞쪽이 한 글자라도 바뀌면 그 뒤 전부 재계산 |
| **순서** | `tools → system → messages` (안 변하는 것 앞, 변하는 것 뒤) |
| **최대 적** | 세션 도중 **모델 전환 · effort 변경 · MCP 추가/제거 · 5분+ 방치** |
| **안전** | 파일 읽기/편집, 스킬·커맨드 호출, `/rewind`, 서브에이전트 spawn |

## 1. 왜 캐시가 필요한가 — 모델은 "기억"하지 못한다

LLM API는 **stateless** 입니다. 턴 사이에 아무것도 기억하지 못하므로, Claude Code는 매 턴 **전체 컨텍스트를 통째로 다시 전송**합니다 (시스템 프롬프트 + 프로젝트 컨텍스트 + 지금까지의 모든 메시지·도구 결과 + 새 메시지). 새 내용은 항상 **끝에 append** 됩니다.

```text
  모델은 stateless → 매 턴 전체를 재전송, 새 내용만 끝에 추가

  턴 1  │ System │ CLAUDE.md │ 대화1 │░신규░│
  턴 2  │ System │ CLAUDE.md │ 대화1 │ 대화2 │░신규░│
  턴 3  │ System │ CLAUDE.md │ 대화1 │ 대화2 │ 대화3 │░신규░│
         └──────────── 매번 동일하게 반복 전송 ─────────┘ └신규┘
```

캐싱이 없으면 턴이 길어질수록 같은 앞부분을 **매번 처음부터 다시 계산**해야 합니다. 100턴짜리 Opus 세션이면 입력 토큰만 $50 ~ $100이 들 수 있습니다. 캐싱은 바로 이 "안 변한 앞부분"의 재계산을 없애는 장치입니다. Anthropic 팀이 *"cache rules everything around me"* 라고 부르며 캐시 적중률을 가동시간(uptime)처럼 모니터링하는 이유입니다.

## 2. 동작 원리 — Prefix(접두) 매칭

캐시는 **요청의 시작부터 일치하는 부분(prefix)** 을 직전에 처리한 내용과 대조합니다. 핵심 규칙은 3가지입니다.

**① 정확 일치 — 의미가 아니라 바이트 단위로 똑같아야 함**

```text
              prefix (앞쪽이 동일) ───────────►│◄─ 뒤쪽
  정상 턴   [ A ][ B ][ C ][ D ] ........... [ 신규 ]
             └──────── 캐시 HIT (재사용·쌈) ───┘   └ 처리·기록 ┘
```

**② 캐시 읽기는 싸고(0.1x), 새 부분만 처리 후 캐시에 기록(write)**

```text
        ┌──────────────── 이번 턴 요청 전체 ────────────────┐
  턴 N  │  System  │  CLAUDE.md  │   과거 대화   │  신규 입력  │
        └──────────┴─────────────┴──────────────┴────────────┘
        ╚═══════ 캐시에서 READ (표준가의 10%) ═══════╝ ╚ 새로 계산 ╝
                                                       ╚ + 캐시 WRITE ╝
```

**③ 앞이 바뀌면 그 뒤 전부 무효 (cascade) — 부분/파일 단위 캐시는 없음**

```text
  B를 한 글자라도 바꾸면?
  [ A ][ B'][ C ][ D ] ........... [ 신규 ]
    │    └──────────── 전부 재계산 (cache MISS) ───────────┘
    └ A만 HIT

  → "B 이후 모든 토큰"이 무효화. 의미가 같아도, 대문자 2글자만 달라도.
    (한 실험: 대문자 2글자 차이로 2,727 토큰 계산이 통째로 폐기됨)
```

요청 안에서 캐시 대상은 **엄격한 순서**로 쌓입니다. 이 순서가 "안 변하는 것을 앞에" 두는 전략의 토대입니다.

```text
  ┌─────────┐   ┌─────────┐   ┌──────────┐
  │  tools  │ → │ system  │ → │ messages │     앞 → 뒤
  └─────────┘   └─────────┘   └──────────┘
   도구 정의      시스템         대화 내용
   (거의 고정)    프롬프트       (매 턴 증가)

  왼쪽을 바꾸면 → 오른쪽 전부 무효. 오른쪽만 늘면 → 왼쪽은 캐시 유지.
```

> 🔧 **기술 디테일(API 레벨):** 명시적 캐시 분기점(breakpoint)은 요청당 **최대 4개**, 캐시 탐색은 분기점에서 뒤로 **최대 20블록** 되짚어 일치하는 prefix를 찾습니다. Claude Code는 이 분기점을 자동 배치하므로 직접 신경 쓸 필요는 없습니다.

## 3. Claude Code의 3계층 구조

Claude Code는 prefix 매칭을 극대화하려고 요청을 "변경 빈도가 낮은 것 → 높은 것" 순서의 3계층으로 구성합니다.

```text
  요청 앞쪽 (거의 안 변함) ───────────────────► 요청 뒤쪽 (매 턴 변함)
  ┌────────────────┬───────────────────┬────────────────────────┐
  │ ① System       │ ② Project         │ ③ Conversation         │
  │    프롬프트     │    컨텍스트        │    (대화)               │
  ├────────────────┼───────────────────┼────────────────────────┤
  │ 코어 지시,      │ CLAUDE.md,        │ 내 메시지,              │
  │ 도구 정의,      │ auto memory,      │ Claude 응답,            │
  │ output style    │ unscoped rules    │ 도구 결과               │
  ├────────────────┼───────────────────┼────────────────────────┤
  │ 변경 시점:      │ 변경 시점:         │ 변경 시점:              │
  │ 도구 집합 변경, │ 세션 시작,         │ 매 턴 (정상)            │
  │ Claude 업그레이드│ /clear · /compact │                         │
  └────────────────┴───────────────────┴────────────────────────┘
        ③만 바뀜  → ①② 캐시 유지 ✅ (정상적인 턴)
        ②를 바꿈  → ②③ 무효 ⚠️
        ①을 바꿈  → ①②③ 전부 무효 🔴 (가장 비쌈)
```

추가로, **프롬프트 텍스트에는 없지만 캐시 키에 포함되는 2가지** 가 있습니다.

| 요소 | 효과 |
| --- | --- |
| **모델(model)** | 모델마다 캐시가 분리됨. `/model` 전환 = 전체 재계산 |
| **effort 레벨** | 같은 모델이라도 effort별로 캐시 분리. `/effort` 변경 = 전체 재계산 (Claude Code가 확인 다이얼로그를 띄움) |

## 4. 비용 구조 — 읽기는 1/10, 쓰기는 1.25 ~ 2배

표준 입력가를 `1.0x` 로 놓았을 때 (예시는 Opus 4.8, 표준 입력 $5/MTok):

```text
  캐시 읽기 (HIT)   0.10x  █                       $0.50/MTok   ⭐ 90% 할인
  표준 입력         1.00x  ██████████              $5.00/MTok
  5분 캐시 쓰기     1.25x  ████████████▌           $6.25/MTok
  1시간 캐시 쓰기   2.00x  ████████████████████    $10.00/MTok
```

| 구분 | 배율 | Opus 4.8 기준 |
| --- | --- | --- |
| 캐시 **읽기** (hit) | **0.10x** | $0.50/MTok |
| 표준 입력 (캐시 미사용) | 1.00x | $5.00/MTok |
| 5분 TTL **쓰기** | 1.25x | $6.25/MTok |
| 1시간 TTL **쓰기** | 2.00x | $10.00/MTok |

**손익분기:** 쓰기는 비싸지만 **1회성** 이고, 읽기는 매 턴 **반복** 됩니다.

- 5분 TTL: 같은 prefix를 **2번만** 재사용해도 이득
- 1시간 TTL: 쓰기가 2배라 같은 시간 안에 **3번 이상** 읽어야 5분 TTL보다 유리

> 💰 써드파티 가이드 추정치(미검증): 잘 관리된 세션에서 캐싱은 하루 입력 토큰 비용의 **40 ~ 50%** 를 절감하며, 위 100턴 Opus 세션($50 ~ $100)이 적중률 90%면 **$10 ~ $19** 수준으로 떨어진다고 보고됩니다.

## 5. TTL — 캐시 수명 (5분 vs 1시간)

캐시는 **마지막 적중 후 일정 시간(TTL)** 안에 다시 쓰지 않으면 만료됩니다. 적중할 때마다 타이머가 리셋되므로, 작업을 계속하는 한 캐시는 "warm" 하게 유지됩니다.

```text
  시간 ───────────────────────────────────────────────────►
  요청 ●────●───●────●─────────( 공백 > TTL )──────────●
       │    │   │    │                                  │
       └ warm warm warm warm ┘                       ❄ COLD
            캐시 읽기 (쌈)                       전체 재계산 (비쌈)

  5분 TTL  : 잠깐만 한눈팔아도 cold       → API key 기본값
  1시간 TTL: 점심·회의 다녀와도 warm 유지 → 구독 기본값
```

| 인증 방식 | 기본 TTL | 비고 |
| --- | --- | --- |
| **Claude 구독** (Pro/Max/Team) | **1시간** (자동) | 토큰 과금이 아니라 플랜에 포함 → 1시간 TTL이 **공짜**. 단, 사용량 한도 초과로 usage credits를 쓰는 중이면 자동으로 5분으로 낮춤 |
| **API 키 / Bedrock / Vertex 등** | **5분** | 토큰당 과금이라 더 싼 5분이 기본. 1시간은 옵트인 |

**관련 환경변수:**

| 환경변수 | 효과 |
| --- | --- |
| `ENABLE_PROMPT_CACHING_1H=1` | API 키 등에서 1시간 TTL 옵트인 |
| `FORCE_PROMPT_CACHING_5M=1` | 인증 방식과 무관하게 5분 강제 (디버깅·비교용) |
| `DISABLE_PROMPT_CACHING=1` | 캐싱 완전 비활성화 (디버깅용). `_HAIKU`/`_SONNET`/`_OPUS`/`_FABLE` 접미사로 모델별 비활성화도 가능 |

## 6. 캐시를 깨뜨리는 것 vs 안전한 것

Claude Code 공식 문서가 가장 가치 있게 정리한 부분입니다. 모두 **세션 도중** 의 행동 기준입니다.

### 🔴 캐시를 무효화하는 행동 (다음 턴이 한 번 느려지고 비싸짐)

| 행동 | 이유 |
| --- | --- |
| **모델 전환** (`/model`) | 모델마다 캐시 분리 → 전체 재계산 |
| **effort 레벨 변경** (`/effort`) | effort도 캐시 키 → 전체 재계산 (확인 다이얼로그) |
| **fast 모드 켜기** | 캐시 키 헤더 추가. 단 **대화당 1회** 만 비용 발생 (이후 토글은 안전) |
| **MCP 서버 연결/해제** | 단, **deferred(지연 로딩) 도구는 안전** ↓ 아래 설명 |
| **플러그인 on/off** | MCP 서버를 제공하는 플러그인일 때만 해당 (스킬·커맨드·훅·테마 제공은 안전) |
| **도구 전체 차단** (deny 규칙에 `Bash` 같은 맨 이름) | 도구 정의가 시스템 계층에 있어 무효화 |
| **`/compact`** | 대화 계층을 요약본으로 교체 → 대화 계층 무효 (의도된 동작) |
| **Claude Code 업그레이드** | 시스템 프롬프트/도구 정의가 바뀜. **세션 resume 후 첫 턴** 이 특히 비쌈 |

> 🔌 **MCP의 핵심 뉘앙스:** 지원 모델에서 기본값인 **deferred tools(tool search로 지연 로딩)** 는 prefix 밖에 있어서, 서버를 붙이거나 떼도 **캐시가 안 깨집니다**. 반대로 tool search가 비활성(Haiku, Vertex AI, 커스텀 `ANTHROPIC_BASE_URL` 게이트웨이)이거나 `alwaysLoad`로 prefix에 올라간 도구는 변경 시 무효화됩니다. stdio 서버 프로세스가 죽거나 HTTP 세션이 만료돼 **자동 재연결** 될 때도 (prefix 로딩 방식이면) 깨질 수 있습니다.

### ✅ 캐시를 유지하는 행동 (안전)

| 행동 | 이유 |
| --- | --- |
| **저장소 파일 편집/읽기** | 읽기는 대화 끝에 append. 편집 시엔 `<system-reminder>` 만 덧붙고 과거 읽기를 소급 변경하지 않음 |
| **CLAUDE.md 세션 중 편집** | 캐시는 안 깨지지만 **변경도 적용 안 됨** ⚠️ (세션 시작 시 로드본 유지, 다음 `/clear`·`/compact`·재시작 때 반영) |
| **output style 변경** | 위와 동일 — 캐시 유지하되 다음 `/clear`·재시작 전까지 적용 안 됨 |
| **권한 모드 변경** | 시스템 프롬프트/도구를 안 건드림 (단 `opusplan`의 플랜 모드는 모델 전환이라 예외) |
| **스킬·커맨드 호출** | 호출 시점에 user 메시지로 주입 → 앞쪽 불변 |
| **`/recap`** | 요약을 커맨드 출력으로 append (`/compact`와 달리 히스토리 교체 안 함) |
| **`/rewind`** | 과거 턴으로 잘라냄 → 그 시점의 캐시 엔트리에 그대로 적중 |
| **서브에이전트 spawn** | 부모 입장에선 호출/결과가 대화에 append될 뿐, 부모 prefix 불변 |

### 모델/effort 전환이 비싼 이유

```text
  같은 텍스트라도 모델마다 캐시가 분리됨 (KV 캐시는 모델 가중치에 종속)

    Opus   ▶ [■■■■ Opus 캐시 ■■■■]
    Sonnet ▶ [■■■■ Sonnet 캐시 ■■■■]   ◄─ /model 로 전환하면
    Haiku  ▶ [■■■■ Haiku 캐시 ■■■■]        전체를 처음부터 재계산

  → opusplan 설정은 플랜 모드(Opus)↔실행(Sonnet)을 오갈 때마다 모델 전환 발생
```

> 💡 **Anthropic의 설계 교훈:** Claude Code는 이 제약을 우회하려고 ⓐ 정보 갱신을 시스템 프롬프트 수정 대신 **`<system-reminder>` 태그** 로 user 메시지에 실어 보내고, ⓑ 플랜 모드를 "도구 집합 변경"이 아니라 **호출 가능한 도구**(EnterPlanMode/ExitPlanMode)로 구현하고, ⓒ 안 쓰는 도구를 제거하지 않고 **deferred 스텁** 으로 두며, ⓓ `/compact` 요약도 부모와 **같은 prefix를 공유** 하는 호출로 만들어 캐시를 재사용합니다.

## 7. 캐시 범위 (Scope) — 머신 + 디렉토리 단위

```text
  시스템 프롬프트에 작업 디렉토리·플랫폼·셸·OS 버전·메모리 경로가 박혀 있음
  → 디렉토리가 다르면 prefix 가 달라져 서로의 캐시를 못 씀

  ~/projA  세션 A ─┐
                   ├─ 같은 디렉토리 + 병렬 실행  → 서로 캐시 공유 ✅
  ~/projA  세션 B ─┘
  ~/projA-worktree 세션 C  → 워크트리는 디렉토리가 달라 별도 캐시 ❌
  ~/projB  세션 D          → 당연히 별도 캐시 ❌
```

- **병렬** 세션(같은 디렉토리)은 prefix가 일치해 서로의 캐시를 읽습니다.
- **순차** 세션은 시작 시점의 git 상태 스냅샷(브랜치·최근 커밋)까지 같아야 공유됩니다.
- **워크트리** 는 각자 작업 디렉토리가 달라 캐시를 공유하지 않습니다.
- API 캐시 자체는 조직(2026-02-05부터는 일부 제공자에서 워크스페이스) 단위로 격리됩니다.

## 8. 모니터링 — 캐시가 일하고 있는지 확인

API 응답의 `usage`에 매 턴 두 숫자가 찍힙니다.

| 필드 | 의미 |
| --- | --- |
| `cache_read_input_tokens` | 캐시에서 읽은 토큰 (표준가의 ~10%로 과금) — **클수록 좋음** |
| `cache_creation_input_tokens` | 이번 턴에 캐시에 새로 쓴 토큰 (쓰기 단가) |

```text
  건강한 세션:  read ████████████████████  creation ██   (읽기 ≫ 쓰기)
  문제 있는 세션: read ██  creation ████████████████████  (매 턴 쓰기가 큼)
                  → prefix 가 계속 바뀌고 있다는 신호. 6장의 무효화 원인 점검
```

확인 방법:

- **`/context`** : 컨텍스트 토큰을 출처별(히스토리·도구 결과·시스템 프롬프트 등)로 분해해 보여줌 → 낭비 지점 발견
- **statusline 스크립트**: `current_usage` 객체를 읽어 실시간 표시
- **OpenTelemetry exporter**: 조직 차원에서 사용자·세션별 캐시 읽기/쓰기 토큰 집계
- 캐시 읽기가 갑자기 0이 되면 "헬스체크 실패"처럼 다뤄야 합니다.

## 9. 서브에이전트와 캐시

```text
  부모 대화 ────────────────────────────────►  (부모 prefix 불변, 캐시 유지)
              │ spawn                    ▲ 결과 append
              ▼                          │
        서브에이전트  [독립 시스템 프롬프트·도구·캐시]
        - 자기만의 캐시를 처음부터 warm-up
        - 구독이라도 서브에이전트는 5분 TTL 사용

  fork (대화 복제): 부모의 시스템 프롬프트·도구·히스토리를 그대로 상속
                    → 첫 요청이 부모 캐시에 적중 ✅
```

- 일반 **서브에이전트**: 부모와 별개의 대화·캐시. 첫 호출은 캐시 미적중에서 시작해 자기 턴들에서 데워짐. 구독이어도 5분 TTL.
- **fork**: 부모 prefix를 그대로 물려받아 첫 요청부터 부모 캐시를 읽음 (`/compact` 요약 호출도 이 방식).
- **활용 팁:** 출력이 큰 작업(테스트 전체 실행, 대량 grep)은 서브에이전트에 위임해 결과 요약만 받으면, 수만 토큰이 메인 대화(=매 턴 재전송되는 prefix)에 쌓이는 것을 막습니다.

## 10. 실전 최적화 체크리스트

공식 문서의 권고 + 써드파티 가이드의 실전 팁을 종합했습니다. (퍼센트 수치는 써드파티 추정치이며 미검증)

### 🟢 세션 시작 시 한 번에 결정 (가장 중요)

- **모델과 effort를 시작할 때 정하고 작업 중엔 바꾸지 않기.** 중간 전환이 가장 흔한 캐시 파괴 원인.
- **필요한 MCP 서버·스킬·플러그인을 미리 켜기.** 작업 도중 "어, 이 도구 없네" 하고 붙이지 않기.
- **CLAUDE.md를 먼저 확정.** 세션 중 편집해도 어차피 적용 안 됨.

### 🟢 작업 중 컨텍스트 관리

- **`/compact` 는 작업 사이 자연스러운 구분점에서** 직접 실행 (자동 트리거를 기다리지 말고). **캐시가 warm 할 때(마지막 메시지 후 ~5분 내)** 하면 요약 호출이 캐시를 재사용해 저렴.
- 다른 작업으로 넘어가면 **`/clear`**. 무관한 토큰을 매 턴 재전송하는 낭비 방지. (방치로 캐시가 cold 됐다면 `/compact`보다 `/clear`가 쌈)
- 버릴 경로로 빠졌으면 `/compact`보다 **`/rewind`** — 이미 캐시된 과거 prefix로 되돌아감.
- **`/context` 로 주기적으로 점검** — 큰 grep 결과 등 낭비 토큰을 찾아 정리.

### 🟢 설정·습관

- **`MAX_THINKING_TOKENS=10000`** — 일상 작업에 thinking 토큰을 상한. (써드파티 추정 30 ~ 40% 절감; thinking도 다음 턴엔 입력이 됨)
- **Sonnet을 기본**, Opus는 복잡한 추론에만. 단순 작업은 `/effort low`.
- **CLAUDE.md는 작고 안정적으로.** 지식 덤프가 아니라 상세 파일을 가리키는 **인덱스** 로. (가이드마다 50줄 ~ 500줄 권고가 갈리나, 핵심은 "자주 안 바뀌는 안정된 규칙"만 두는 것)
- **구체적으로 지시.** 파일·함수명을 명시하면 막연한 탐색으로 인한 대량 도구 출력을 줄임. (3턴 왕복 vs 1턴 해결 = 수만 토큰 차이)
- 긴 작업엔 API 키 사용자도 **1시간 TTL**(`ENABLE_PROMPT_CACHING_1H=1`) 고려.

## 11. 알려진 함정

| 함정 | 내용 |
| --- | --- |
| **CLAUDE.md·output style 중간 편집의 "조용한 무시"** | 캐시는 안 깨지지만 **변경도 적용 안 됨.** 바꿨는데 동작이 그대로면 `/clear`·재시작 필요. |
| **세션 resume + 업그레이드** | 업그레이드 후 긴 대화를 resume하면 새 시스템 프롬프트 뒤에 오게 되어 **전체 히스토리를 캐시 없이 재처리** → 그 첫 턴이 세션 전체에서 가장 비쌀 수 있음. |
| **`--resume`/`/resume` 캐시 회귀 (커뮤니티 보고)** | 일부 버전에서 resume 시 attachment 블록 위치가 흩어지고(`cc_version` 핑거프린트 불안정·도구 정렬 비결정성) 캐시가 **조용히** 깨져 시간당 $0.50 → $5 ~ $10 (최대 20배)로 치솟는 사례 보고. 에러 없이 진행돼 알아채기 어려움. (`cache_read`가 0인지 모니터링이 방어선) |
| **opusplan의 플랜 모드 토글** | 플랜↔실행 전환마다 Opus↔Sonnet 모델 전환 = 매번 새 캐시. |
| **자동 재연결 MCP** | prefix 로딩 방식일 때, 내가 아무것도 안 했는데 서버가 죽었다 살아나며 캐시가 깨질 수 있음. |

## 마치며

캐시의 원리는 결국 한 문장으로 압축됩니다. **"안 변하는 것은 앞에, 변하는 것은 뒤에."** 그리고 세션 도중 모델·effort·도구 집합을 건드리지 않는 것. 이 두 가지만 지켜도 대부분의 낭비는 사라집니다. 정말로 캐시가 일하고 있는지 궁금할 땐 `/context`와 `usage`의 `cache_read_input_tokens`를 보면 됩니다 — 읽기가 쓰기를 압도한다면, 캐시는 제 몫을 하고 있는 겁니다.
