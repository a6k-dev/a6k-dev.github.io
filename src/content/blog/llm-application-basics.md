## LLM 애플리케이션의 기본 구조

LLM 애플리케이션은 결국 **"좋은 컨텍스트를 모아 모델에 잘 전달하는 일"** 입니다.
모델 자체는 강력하지만, 우리 데이터를 모르기 때문에 필요한 정보를 어떻게 채워 넣느냐가 품질을 좌우합니다.

```
사용자 질문 → (관련 문서 검색) → 프롬프트 구성 → LLM 호출 → 응답
```

## 1. 컨텍스트 윈도우를 의식하기

모델이 한 번에 볼 수 있는 토큰에는 한계가 있습니다. 긴 문서를 통째로 넣기보다는
**필요한 부분만 추려서** 넣는 전략이 비용과 품질 모두에 유리합니다.

## 2. RAG (Retrieval-Augmented Generation)

사내 문서나 최신 정보처럼 모델이 학습하지 않은 지식을 다룰 때 쓰는 패턴입니다.

1. 문서를 청크(chunk)로 쪼개 **임베딩**으로 변환해 벡터 DB에 저장
2. 질문이 들어오면 의미적으로 가까운 청크를 검색
3. 검색된 청크를 프롬프트에 넣어 모델에게 "이 근거로 답하라"고 지시

```python
chunks = retriever.search(query, top_k=5)
context = "\n\n".join(c.text for c in chunks)
prompt = f"""다음 문서를 근거로 질문에 답하세요.

문서:
{context}

질문: {query}
"""
```

## 3. 도구 사용 (Tool Use)

모델이 직접 계산하거나 외부 API를 호출해야 할 때는 **함수(도구)를 정의**해 모델이
필요할 때 호출하도록 합니다. 모델이 "이 도구를 이런 인자로 부르고 싶다"고 알려주면,
애플리케이션이 실제로 실행한 뒤 결과를 다시 모델에 전달합니다.

## 평가를 빼먹지 말 것

LLM 애플리케이션은 **비결정적**이라 "잘 되는 것 같다"는 느낌만으로는 위험합니다.
대표 질문 세트를 만들어 정답/근거 일치 여부를 꾸준히 측정하는 **평가 파이프라인**을
초기에 갖춰두는 것을 강력히 추천합니다.
